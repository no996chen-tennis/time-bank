# 时间银行 — iOS 媒体存储技术方案

> Version: 1.1 | Date: 2026-04-22 | 配套 PRD V1.3
>
> **V1.1 变更：** 加入 App Group 架构决策（Widget 共享数据方案）；CloudKit 同步拆两阶段（metadata-only → media）；新增 §8 媒体兼容矩阵；修正 Info.plist 所需权限（V1 不需要麦克风权限，只在 V1.5+ 视频拍摄时才加）。

本文回答一个问题：**用户在 App 里"存入"一段时刻（带图片/视频）时，这些媒体到底存在哪里、怎么读、怎么备份、会不会丢？**

---

## 一、方案选型对比

| 方案 | 存储位置 | 钱 | 多设备同步 | 隐私 | 离线 | 实现成本 | 风险 |
|------|---------|----|-----------|------|------|---------|------|
| **A. App 沙盒 + FileManager** | `Documents/` | 0 | ❌ | 最高（不出设备） | ✅ | 低 | 换机丢数据、无备份 |
| **B. App 沙盒 + CloudKit 同步** | 沙盒 + iCloud 私有库 | 0（用户 iCloud 配额） | ✅ | 高（Apple 生态闭环） | ✅ | 中 | 依赖用户开 iCloud |
| **C. 写入系统相册自定义 Album** | Photos.app | 0 | ✅（iCloud 照片自动） | 中（用户可误删） | ✅ | 低 | 数据和相册混在一起、需读写相册权限 |
| **D. 自建后端 + 对象存储（S3/OSS）** | 云服务 | 有成本 | ✅ | 低 | ❌ | 高 | 账号体系、合规、成本不可控 |

### 结论：V1 用 A，V1.5 升级到 B，**不选 C / 不选 D**

**为什么不选 C（写相册）：**
- 用户相册本身已经信息过载，再写一份进去会让用户困惑
- "时间银行里的时刻"应该是有强归属（属于某个维度、有上下文文字）的，跟相册流水不是同一种东西
- 用户删相册照片时可能误删我们的媒体，反而降低"永久封存"的承诺感
- 需要 `NSPhotoLibraryAddUsageDescription`，上架审核要多写理由

**为什么不选 D（自建云）：**
- 违反 PRD 的 Non-Goals（V1 不做账号体系、不做订阅）
- 隐私上用户要"把家庭照片上传到一个陌生人服务器"，心理门槛高
- 每月带宽和存储费用会随用户增长线性上涨，早期没收入扛不住

## 二、V1 方案：App 沙盒 + SwiftData

### 2.1 文件组织

```
App Sandbox/
└── Documents/
    └── TimeBank/
        └── moments/
            ├── 8f3a-…-uuid-1/          ← 一个 Moment 一个文件夹
            │   ├── 01.heic             ← 原图（iOS 默认格式）
            │   ├── 01.thumb.jpg        ← 200x200 缩略图（加速列表渲染）
            │   ├── 02.heic
            │   ├── 02.thumb.jpg
            │   └── 03.mov              ← 原视频
            │       ├── 03.thumb.jpg    ← 视频首帧缩略图
            │       └── 03.poster.jpg   ← 视频封面（可选）
            └── 9b2c-…-uuid-2/
                └── 01.heic
```

**为什么不把文件直接平铺：**
一条时刻一个文件夹，删除时刻只要递归删这个文件夹；导出时把文件夹打包成 ZIP 子目录即可；出故障也容易定位。

### 2.2 数据模型（SwiftData）

```swift
import SwiftData
import Foundation

@Model
final class Moment {
    @Attribute(.unique) var id: UUID
    var dimensionId: String       // "parents" / "kids" / custom UUID
    var title: String?
    var note: String
    var happenedAt: Date
    var durationSeconds: Int?
    var createdAt: Date
    var updatedAt: Date

    @Relationship(deleteRule: .cascade, inverse: \MediaItem.moment)
    var mediaItems: [MediaItem] = []

    init(dimensionId: String, happenedAt: Date) {
        self.id = UUID()
        self.dimensionId = dimensionId
        self.happenedAt = happenedAt
        self.note = ""
        self.createdAt = .now
        self.updatedAt = .now
    }
}

@Model
final class MediaItem {
    @Attribute(.unique) var id: UUID
    var typeRaw: String           // "image" / "video"
    var relativePath: String      // "moments/{moment-uuid}/01.heic"
    var thumbnailPath: String?
    var fileSize: Int64
    var durationSeconds: Int?     // 视频时长
    var sortIndex: Int
    var createdAt: Date
    var moment: Moment?

    var type: MediaType { MediaType(rawValue: typeRaw) ?? .image }
}

enum MediaType: String { case image, video }
```

**关键点：**
- `relativePath` 存**相对** `Documents/` 的路径，**不存绝对路径**。iOS 升级或迁移时沙盒绝对路径会变，相对路径才稳定。
- `deleteRule: .cascade` 让删 Moment 时 SwiftData 自动删 MediaItem 记录。但**物理文件不会自动删**，需要在自定义的删除函数里手动递归删文件夹（见 2.5）。
- 文件大小、缩略图路径这些都存元数据，列表渲染时不用去摸文件系统。

### 2.3 写入流程（新增时刻）

```swift
// 伪代码
func saveMoment(
    dimensionId: String,
    happenedAt: Date,
    duration: TimeInterval?,
    title: String?,
    note: String,
    pickedItems: [PhotosPickerItem]   // 来自 PhotosPicker
) async throws {
    // 1. 建 Moment 实体
    let moment = Moment(dimensionId: dimensionId, happenedAt: happenedAt)
    moment.title = title
    moment.note = note
    moment.durationSeconds = duration.map { Int($0) }

    // 2. 创建文件夹
    let momentDir = documentsURL
        .appendingPathComponent("TimeBank/moments/\(moment.id.uuidString)")
    try FileManager.default.createDirectory(at: momentDir,
                                            withIntermediateDirectories: true)

    // 3. 遍历 PhotosPicker 拿到的 items
    for (index, item) in pickedItems.enumerated() {
        let (type, ext, data) = try await loadPickedItem(item)
        // 原始媒体
        let fileName = String(format: "%02d.%@", index + 1, ext)
        let fileURL = momentDir.appendingPathComponent(fileName)
        try data.write(to: fileURL)

        // 缩略图
        let thumbName = String(format: "%02d.thumb.jpg", index + 1)
        let thumbURL = momentDir.appendingPathComponent(thumbName)
        try await generateThumbnail(from: fileURL,
                                    type: type,
                                    output: thumbURL,
                                    size: CGSize(width: 400, height: 400))

        // MediaItem 记录
        let media = MediaItem()
        media.id = UUID()
        media.typeRaw = type.rawValue
        media.relativePath = "TimeBank/moments/\(moment.id.uuidString)/\(fileName)"
        media.thumbnailPath = "TimeBank/moments/\(moment.id.uuidString)/\(thumbName)"
        media.fileSize = Int64(data.count)
        media.sortIndex = index
        media.createdAt = .now
        moment.mediaItems.append(media)
    }

    // 4. 存库
    modelContext.insert(moment)
    try modelContext.save()
}
```

**缩略图生成：**
- 图片：`CGImageSource` + `kCGImageSourceCreateThumbnailFromImageAlways` —— 比加载整图再缩放快 10x
- 视频：`AVAssetImageGenerator` 抽取 0.1s 处的一帧

**HEIC/HEVC 保留：** 不要重编码成 JPG/MP4，会损失画质和空间。iOS 从 iOS 11 起原生支持。

### 2.4 读取流程（时间线 / 详情页）

**时间线列表：** 只加载 `thumbnailPath` 对应的缩略图，原图不碰。SwiftUI 用 `AsyncImage` 或 `ImageRenderer`。
**详情页：** 按需加载原图/原视频，视频用 `AVPlayer` 播放本地文件 URL。

```swift
extension MediaItem {
    var fileURL: URL { documentsURL.appendingPathComponent(relativePath) }
    var thumbURL: URL? {
        thumbnailPath.map { documentsURL.appendingPathComponent($0) }
    }
}
```

### 2.5 删除流程（CRITICAL — 常见坑）

SwiftData 的 cascade 只删数据库记录，**物理文件需要手动删**。

```swift
func deleteMoment(_ moment: Moment) throws {
    let momentDir = documentsURL
        .appendingPathComponent("TimeBank/moments/\(moment.id.uuidString)")
    try? FileManager.default.removeItem(at: momentDir)  // 先删文件
    modelContext.delete(moment)                          // 再删记录
    try modelContext.save()
}
```

**保险做法**：App 启动时跑一次"孤儿文件扫描"——遍历 `moments/` 下的文件夹，如果 UUID 不在数据库里，清理掉。

### 2.6 容量预估

| 单位 | 平均体积 |
|------|---------|
| 1 张 HEIC 图（12MP iPhone） | 1.5-3 MB |
| 1 段 10s HEVC 视频（1080p） | 15-25 MB |
| **1 条时刻（3 图 + 1 短视频）** | **~35 MB** |
| **1000 条时刻** | **~35 GB** |

iPhone 起步 128GB，存 1000 条远超用户实际需求（按每周存 2 条，10 年才 1000 条）。但要：
- 设置页显示"当前占用 X GB"
- 提供"清理缩略图缓存"选项（重新生成即可）
- 视频上传时提示"大文件占 X MB，确认？"

### 2.7 权限 & 隐私（V1 最小权限策略）

| 动作 | V1 | V1.5+ |
|------|-----|------|
| 从相册选图/视频 | ✅ 通过 `PhotosPicker` (iOS 16+)，**无需任何权限** | 同 |
| App 内拍照 | ❌ V1 不做 | ✅ 加 `NSCameraUsageDescription` |
| App 内录视频 | ❌ V1 不做 | ✅ 加 `NSCameraUsageDescription` + `NSMicrophoneUsageDescription` |
| 写入自己沙盒 | ❌ 不需要权限 | 同 |
| 读系统相册 | ❌ 不做（用 PhotosPicker 替代） | 同 |
| 写入系统相册 | ❌ V1 不做（保持"不碰用户相册"主张） | V1.5 再议 |
| 通知 | ✅ `UserNotifications` 请求权限 | 同 |

**V1 的 Info.plist 权限块应该只有：**
```
（V1 不需要任何 Camera / Microphone / PhotoLibrary 描述 key）
```

**这是一条营销卖点**：商店描述可以明说"V1 不申请任何敏感权限"。V1.5 加上相机/麦克风时，Info.plist 才相应加：

```
<key>NSCameraUsageDescription</key>
<string>用来拍一段当下，存进你的时间银行</string>

<key>NSMicrophoneUsageDescription</key>
<string>视频录制时需要，让你能听见当时的声音</string>
```

## 三、V1.5 方案：分两阶段上 CloudKit 同步（关键架构决策）

### 3.1 为什么要分两阶段

V1.5 同步看似"SwiftData + CloudKit 一键开启"，但实际上**原媒体同步**是个独立大工程：

- 原图 HEIC 平均 2MB，视频可达 50MB+
- 全量同步会炸用户带宽 + iCloud 配额
- 冲突处理、断点续传、失败重试、跨设备懒加载、孤儿 CKAsset 清理 —— 每个都是独立模块
- 一上来全做，V1.5 会拖成 V2.0

**所以 V1.5 拆成两个子版本 A / B：**

### 3.2 V1.5-A：Metadata-Only Sync（先做这一版）

**目标：** 跨设备能秒级看到同样的 Moment 列表 + 元信息 + 缩略图，但原媒体还锁在单台设备上。

**范围：**
- ✅ SwiftData 全量模型同步：UserProfile / Dimension / Moment / MediaItem 元数据 / Settings
- ✅ **缩略图**同步（200×200 JPG，平均 20-50KB，1000 Moment 也只 20-50MB）
- ❌ 原图/原视频**不同步**

**UX 影响：**
- 跨设备打开时间线：顺畅（看的是缩略图）
- 跨设备点开时刻详情，原图在另一台设备上：显示缩略图放大版 + 标签"这张原图在你的其他设备上"
- 提供按钮「从另一台设备取回」→ V1.5-B 才做

**工程难度：** 中。~ 2 周。

**可发布条件：**
- [ ] 跨两台设备双向同步 50+ Moment 无冲突
- [ ] iCloud 不可用时自动降级本地模式，banner 提示
- [ ] 新装 App 登录 iCloud 后首次同步 ≤ 60s

### 3.3 V1.5-B：Media Sync（后做这一版，视用户反馈决定优先级）

**目标：** 真正的全量同步。用户在任何设备都能看到并拉回原图原视频。

**范围：**
- ✅ 原图 / 原视频的**懒加载 CKAsset 同步**
- ✅ 断点续传、后台下载队列
- ✅ 单设备存储空间监控（避免同步一半撑满手机）
- ✅ 冲突处理（同一 Moment 在两设备同时编辑，保留两份以 `-v2` 后缀）

**UX 影响：**
- 时刻详情："从 iCloud 取回原图（3.2MB）" → 按钮
- 设置页：「同步时下载原媒体」toggle（默认关，用户开启后静默后台下载）
- 提供"清理本地原媒体缓存"（只保留缩略图 + 云端原文件）

**工程难度：** 高。~ 3-4 周。Apple 官方示例对 CKAsset 懒加载的支持不够完整，很多坑要自己踩。

### 3.4 失败兜底（两阶段通用）

- iCloud 未登录：降级成 V1 纯本地模式，banner 提示"开启 iCloud 可防止换机丢失"
- 同步失败：保留本地数据为准，后台静默重试 + 指数退避
- 账户切换：SwiftData + CloudKit 自动处理（会询问用户是否合并）
- 配额不足：Alert + 提示升级 iCloud 存储

## 三·半、App Group 架构（V1 就必须做）

**为什么 V1 就要做：** Widget Extension 是独立进程，不能直接访问主 App 的 SwiftData 数据库。必须在 Day 1 就定好架构，后改成本高。

### 3½.1 App Group 配置

- **Group ID**：`group.com.adamchen.timebank`
- Xcode → 项目 → Signing & Capabilities → + Capability → App Groups → 勾选主 App target + Widget Extension target

### 3½.2 SwiftData 位置决策

**不把 SwiftData 数据库放 App Group**，原因：
1. SwiftData 不是为多进程设计的，并发写会有问题
2. 读写冲突会导致主 App 启动时数据库损坏
3. Widget 不需要写入任何数据

**正确做法：** SwiftData 留在主 App 沙盒默认位置（`~/Library/Application Support/...`），主 App 在关键数据变更时派生一份 **Widget 专用 snapshot JSON** 放到 App Group 共享容器。

### 3½.3 Widget Snapshot 结构

```
// App Group Container / snapshot.widget.json
{
  "version": 1,
  "generatedAt": "2026-04-22T09:00:00Z",
  "totalHours": 128,
  "totalMoments": 27,
  "dimensions": [
    {
      "id": "parents",
      "name": "陪父母",
      "iconKey": "heart",
      "colorKey": "rose",
      "consumeHours": 552,
      "storedHours": 56,
      "momentCount": 12,
      "mode": "normal",
      "lastMoment": {
        "title": "杭州西湖 48 小时",
        "happenedAt": "2026-03-22T10:00:00Z",
        "daysAgo": 30
      }
    },
    // ... 其他 visible 维度
  ],
  "topText": "你和爸妈还能一起度过约 552h",
  "topTextType": "neutral"
}
```

### 3½.4 Snapshot 写入触发点

主 App 在以下场景写一次 snapshot.widget.json：
- 首次启动完成 onboarding 后
- 新增 / 编辑 / 删除（提交后）一条 Moment 后
- 改维度参数后
- Settings 变更（文案风格、偏好维度）后
- 每次 App 进入前台时（如果距上次写入 > 1 小时）
- 凌晨 4 点（如果 App 被后台唤起）

### 3½.5 Widget 读取规则

```swift
// WidgetKit TimelineProvider
func getTimeline(in context: Context, completion: @escaping (Timeline<Entry>) -> Void) {
    let groupURL = FileManager.default.containerURL(
        forSecurityApplicationGroupIdentifier: "group.com.adamchen.timebank"
    )!
    let snapshotURL = groupURL.appendingPathComponent("snapshot.widget.json")
    let data = try? Data(contentsOf: snapshotURL)
    let snapshot = try? JSONDecoder().decode(WidgetSnapshot.self, from: data)

    // 若 snapshot 不存在或过期 > 72h → 显示降级文案
    let entry = Entry(snapshot: snapshot ?? .placeholder, date: .now)
    completion(Timeline(entries: [entry], policy: .atEnd))
}
```

## 八、媒体兼容矩阵（V1 真机测试必查表）

这张表是开发时要**实测的兼容性清单**。每个类型都要在真机上至少走一次 "PhotosPicker 选入 → 保存 → 缩略图生成 → 详情页查看" 的闭环。

### 8.1 图片类

| 格式 / 类型 | 支持？ | 存储做法 | 备注 |
|-----------|-------|---------|------|
| HEIC（iPhone 默认） | ✅ | 保留原格式 | iOS 11+ 原生 |
| JPG / JPEG | ✅ | 保留原格式 | |
| PNG | ✅ | 保留原格式 | |
| HEIF | ✅ | 保留原格式 | |
| GIF | ⚠️ | 保留但详情页只展示静态首帧，不播放动画 | V1.1 可考虑动图播放 |
| Live Photo | ⚠️ | **只提取静态部分（JPG/HEIC），丢弃动态部分** | V1.5 再做动图还原 |
| HDR (iOS 17+) | ✅ | 保留原格式（PhotosPicker 返回的 Data 已包含 HDR 信息）| 缩略图做 SDR 降级 |
| RAW / DNG | ✅ | 保留原格式 | 缩略图走 CGImageSource |
| 超大尺寸（> 50MB） | ✅ | 保留但写入时显示 ProgressBar | |

### 8.2 视频类

| 格式 / 类型 | 支持？ | 存储做法 | 备注 |
|-----------|-------|---------|------|
| HEVC / H.265（iPhone 默认） | ✅ | 保留原格式 | |
| H.264 | ✅ | 保留原格式 | |
| 慢动作（Slo-mo） | ⚠️ | **存储原视频但播放速度为标准速**（V1 不做变速还原） | V1.5 可考虑 |
| 延时摄影（Time-lapse） | ✅ | 保留原格式，正常播放 | |
| 竖屏 / 横屏 / 旋转过的 | ✅ | 保留原始 AVAsset 的 rotation metadata，`AVPlayer` 自动处理方向 | |
| 4K / 60fps | ✅ | 保留原格式 | 存储体积警告 |
| ProRes（Pro 机型） | ⚠️ | 保留原格式 | 文件非常大，提示用户 |
| 单视频 > 2 分钟 | ❌ | **直接拒绝**，Toast 告知用户 | V1 明确策略 |
| 单视频 > 500MB | ⚠️ | **保存前 Alert 警告**"这个视频 X MB，占手机空间较大。要继续吗？" | |

### 8.3 EXIF & 元数据

V1 策略：**全部保留**。
- 位置信息（GPS）：默认保留，但 V1.1 考虑给用户一个开关"保存前去除位置信息"
- 拍摄时间：保留（但不强制作为 Moment 的 `happenedAt`，用户可自己改）
- 相机型号 / 光圈等：保留，不展示，不分析
- 人物识别 / 人脸 tag：**不读取**

### 8.4 真机测试最低要求

上架前必须在至少**一台 iPhone 真机**上跑通：
- [ ] 从相册选 HEIC 单图 + 写入 + 详情页查看
- [ ] 从相册选 HEIC 单图 + JPG 单图混合 + 写入
- [ ] 从相册选 HEVC 10s 视频 + 写入 + 详情页播放
- [ ] 从相册选 Live Photo + 验证只存了静态
- [ ] 从相册选 > 2 分钟视频 + 验证被拒绝
- [ ] 从相册选 9 张满 + 验证第 10 张无法选中
- [ ] 写入过程中按 Home 键 + 再回到 App + 验证状态一致

## 四、导出功能（P0-12 实现思路）

用户在设置页点"导出全部数据" → App 执行：

1. 创建临时目录 `tmp/export-{yyyyMMdd-HHmmss}/`
2. 按维度拆分子目录 `parents/ / kids/ / create/ / ...`
3. 每个时刻一个子文件夹 `2024-03-15-杭州西湖/`
   - 复制所有原图原视频进去（保持 HEIC/HEVC 或转 JPG/MP4 由用户选）
   - 写一个 `moment.json`（元数据：标题、时长、文字、发生时间）
   - 同时写一个 `README.md`（人类可读版，万一 App 没了用户用记事本也能看）
4. `ZipArchive` 或系统 `NSFileCoordinator` 打包
5. 用 `UIActivityViewController` 让用户"分享到 文件 / AirDrop / 邮件"

**大招：**
也可以额外导出一个纯 Markdown 版本，每条时刻一个 `.md` 文件，图片用相对路径引用 —— 用户把 ZIP 解压到 Obsidian 里就是一份现成的"时间银行笔记库"。这对有笔记习惯的用户是巨大加分项。

## 五、开发里程碑建议

| 阶段 | 工作量 | 产出 |
|------|-------|------|
| **M1 - 本地存储闭环** | 1 周 | 能新增时刻（图/文）、看时间线、看详情、删除。纯本地。 |
| **M2 - 视频 + 缩略图** | 3 天 | 支持视频上传、自动生成缩略图、视频播放器。 |
| **M3 - 导出功能** | 3 天 | ZIP 导出 + Markdown 导出。 |
| **M4 - CloudKit 同步** | 5 天 | 开 CloudKit、测多设备、写兜底。放 V1.5 上线即可。 |

## 六、关键风险 & 应对

| 风险 | 应对 |
|------|------|
| 用户传了 20 张大图，写盘卡顿 | 异步写入 + 前台 ProgressBar，禁止期间退出页面 |
| 视频太大超 500MB | 限制单视频 ≤ 2 分钟，或自动压缩到 1080p 30fps |
| iCloud 空间不够 | 降级本地，明确提示用户 |
| SwiftData 数据迁移 | 所有 @Model 新增字段必须有默认值，避免升级后旧数据解析失败 |
| 用户误删 App | 导出功能 + iCloud 同步双保险，不同步的用户首次打开时有弹窗建议开 iCloud |

## 七、最后一句

**V1 做 A 方案就够了**——简单、可控、隐私最好。把 CloudKit 留给 V1.5，避开 App Review 上可能因 iCloud 配置问题被拒的风险。等用户到 100+ 再上同步，那时你的测试用户也多了，问题能更快暴露。
