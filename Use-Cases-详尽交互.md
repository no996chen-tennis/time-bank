## 0. 通用约定

### 0.1 页面层级命名

| 层级 | 说明 | 交互性质 |
|------|------|---------|
| **页面 (Screen)** | 独立导航栈一层，可返回 | 全屏切换 |
| **模态 (Modal)** | 从底部弹出，遮盖当前页 | 全屏遮罩 |
| **面板 (Sheet)** | 半屏弹层，可拖拽 | 半屏遮罩 |
| **弹窗 (Alert)** | 系统原生确认框 | 居中阻塞 |
| **Toast** | 顶部或底部提示条，自动消失 | 非阻塞 |
| **气泡 (Tooltip)** | 锚定在某控件旁，点外面消失 | 非阻塞 |

### 0.2 通用按钮图例

| 图标 | 含义 | 位置 |
|------|------|------|
| `‹` | 返回 | 导航栏左上 |
| `×` | 关闭模态 | 模态左上 |
| `+` | 新增 | 导航栏右上 / 卡片右上 |
| `⋯` | 更多动作 | 导航栏右上 / 详情页 |
| `⚙` | 设置入口 | 主页右上 |

### 0.3 数据原子性要求（V1 强原子性）

涉及多次写入的操作（如"保存时刻"要写 Moment + N 个 MediaItem + N 个文件）必须**严格事务化**：
- 任意一步失败 → **全部回滚**（删除已创建的文件 + 撤销 SwiftData 插入）
- **V1 禁止"部分成功"** —— 例如"5 张图中的第 3 张加载失败"→ 整个保存失败，不允许"跳过失败的那张继续存"
- 只有当用户明确看到"存下了"的 Toast 时，整条 Moment 才算真的存在
- 对一个记忆产品来说，"半成功"会直接破坏用户信任语义

### 0.4 状态机通用规则

- 列表页数据变更后：**局部刷新 + 新卡片带轻微 spring 动画**（不重绘整个列表）
- 删除数据后：**被删项做淡出 + 列表上移** 动画（300ms）
- 新增数据后：**顶部插入 + 淡入向下推动** 动画（300ms）

# UC-0 Onboarding 首次引导

## UC-0.1 首次启动完整流程（V1.2 · 支持单身 / 无孩子 / 已故亲人）

**触发：** 用户在 App Store 下载并首次打开 App

**前置：** 设备无任何本地数据

**设计原则：**
- 引导绝不预设"每个用户都有完整家庭"（单身、丁克、父母离世、丧偶等都常见）
- 任何家人信息都可跳过 + 可回头改
- "没有 X"不是缺失，是一种正常状态——该时间账户直接不展示，不显示占位文案让用户有遗憾感

### 数据变更

- `UserProfile` 写 1 条：生日、性别、以及每个家人结构体（可空）
  - `UserProfile.parents`: { father: {year, deceased}?, mother: {year, deceased}?, visitsPerYear, hoursPerVisit } | null
  - `UserProfile.children`: Array<{year, gender?}> | null
  - `UserProfile.partner`: { year, hoursPerDay } | null
  - `UserProfile.solo`: bool (是否强调独处追踪)
- `Dimension` 写入 6 条内置时间账户，但每条带 `status`：
  - 用户勾选的对应时间账户 → `status = visible`
  - 未勾选的对应时间账户 → `status = hidden`（不展示，但配置已在，将来开启只需切换）
  - 通用时间账户（运动/创造/自由）→ 默认 `visible`
- `Settings` 写入：`narrativeMode = positive` / `notificationEnabled`

### 异常路径

- 生日未填 → 「下一步」置灰
- Step 2 一个 chip 都没选 → 允许进入，最终只展示通用时间账户（运动/创造/自由）+ 今日此刻
- Step 3 某家人信息全留空 → 对应时间账户 `status = hidden`，不展示
- 中途杀进程重启 → 从临时 `OnboardingDraft` 恢复到最后步骤

### 后续修改入口（UC-1 系列）

用户后续可在 设置 → 个人信息 里：
- 添加之前未勾选的家人信息 → 对应时间账户 `status` 转 visible
- 移除家人（UC-1.3）→ 时间账户 `status` 转 hidden；原瞬间保留
- 标记某家人"已故" → 该时间账户不再显示"还能共度"数字，只保留"已存入"时间线（成为纪念型时间账户）

## UC-0.2【U】补充或修改 Onboarding 时跳过的信息

**触发场景：** 用户 Onboarding 时没勾选"有伴侣"，后来结婚了

**入口：** 设置 → 个人信息 → 底部「添加家人信息」

**步骤：**
1. 点入口 → 推入 **添加页**（与 Step 3 单卡片形态相同）
2. 选「伴侣」 → 展开卡片 → 填写
3. 点保存 → `UserProfile.partner = {...}` + `Dimension[id=partner].status = visible`
4. 返回设置页 → 主页下次进入时**新卡片带高亮动画**出现，顶部 Toast"陪伴侣时间账户已加入"

## UC-0.3【特殊】标记某家人"已故"

**触发场景：** 用户经历丧亲。

**入口：** 设置 → 个人信息 → 对应家人详情 → 「标记已故」

**设计哲学：** 不把这个功能藏起来，但要**足够温柔**。

**步骤：**
1. 点「标记已故」 → 推入一个独立的确认页（不是 Alert，因为 Alert 太冷）：
   ```
   想做一个艰难的标记。

   [家人名/称谓] 已经不在了。
   时间银行会把这个时间账户变成
   一个"纪念页"：
    · 不再显示"还能一起度过多少小时"
    · 保留你存入的所有瞬间，永远
    · 顶部会显示"共存入 56 小时，12 个瞬间"

   你以后还能取消标记。

   [不了，再想想]    [确认标记]
   ```
2. 点「确认标记」 →
   - `UserProfile.parents.father.deceased = true` (举例)
   - 若父母均标记已故 → `Dimension[id=parents].mode = memorial`（纪念模式，只展示存储层）
3. 主页对应卡片视觉变化：
   - 保留原时间账户色（不变灰，不变冷）
   - 数字区只显示存储层："已存入 56h · 12 个瞬间"
   - 下方 chip："致爸爸"
   - 点击进入详情：顶部不显示剩余小时计算，直接是时间线

**数据变更：**
- `UserProfile.{家人}.deceased = true` + `deceasedAt = Date`
- 对应 `Dimension.mode = memorial`

**返回：** 导航栏 `‹`

## UC-1.2【U】修改个人信息

**入口：** 个人信息详情页

**步骤：**
1. 在详情页点某一行（如「父亲出生年」）
2. 系统弹出 **Sheet**：
   - 年份字段：`1964`（可输）
   - 确认按钮「保存」
3. 用户修改 `1964 → 1965`，点「保存」
4. Sheet 收起 → 详情页对应行刷新为 `1965（59 岁）`
5. **全局影响**：相关时间账户（此处是「陪父母」）的剩余小时数 **立刻重新计算**，下次进主页看到新数字

**数据变更：**
- `UserProfile.father.birthYear = 1965`
- 所有依赖该字段的 `Dimension.computedRemaining` 标记为 dirty → 主页下次渲染时重算

**异常：**
- 年份 > 今年 → 红字提示"不能设为未来"
- 年份 < 1900 → 红字提示"年份太早了"
- 保存时磁盘满 → Toast"保存失败，手机空间不够"

## UC-1.4【U】调整预期寿命（影响时间余额）· V1.2 新增

**入口：** 设置 → 个人信息 → 「预期寿命」一行 · 或 主页双层卡上半"时间余额"区点击跳转

**步骤：**
1. 点"预期寿命" → 弹底部 Sheet
2. Sheet 内容：
   ```
   调整预期寿命
   ────────────────
   [滑块 60 ─●── 100 岁]  默认 85
   ────────────────
   不用太准，估一下就好。
   这个数字只影响"时间余额"的计算。
   [取消][保存]
   ```
3. 用户拖滑块 → 底部实时预览新的时间余额周数
4. 点"保存" → `UserProfile.expectedLifespanYears = newValue`
5. Sheet 收起 → 主页双层卡上半数字**roll 动画**刷新

**默认值建议：**
- 85（全球平均 + 现代医学进步的保守估算）
- 可调范围：60-100
- 低于"当前年龄 + 5" 时给警告（不拦截保存）

**数据变更：**
- `UserProfile.expectedLifespanYears`
- 所有依赖 lifespan 的 Widget snapshot 重新生成

## UC-1.3【D】删除家人信息

**入口：** 个人信息详情页 → 某个家人卡片

**步骤：**
1. 个人信息详情页长按「孩子」卡片（0.5s） → 弹出底部 **Sheet**：
   - 按钮1：编辑
   - 按钮2：**移除孩子信息**（红字）
   - 按钮3：取消
2. 点「移除孩子信息」 → 系统 **Alert**：
   ```
   移除后，"陪孩子"时间账户会从主页消失。
   你存入这个时间账户的 8 个瞬间会保留到"其他"分类里，
   你随时能把它们找回来。

   [取消]  [移除]
   ```
3. 点「移除」 → 执行**统一时间账户下沉协议**（见下）

### 统一时间账户下沉协议（Dimension Demotion Protocol）

**适用场景：** UC-1.3 移除家人信息 / UC-2.6 删除自定义时间账户 / 其他任何让一个时间账户消失的场景。

**协议：**
1. 该时间账户 `Dimension.status = hidden`（非 deleted，保留原配置以便未来"再加回来"时参数不丢）
2. 所有原属该时间账户的 Moment（`status == normal`）：
   - `originDimensionId = <原时间账户 id>`（**永远记住自己曾经属于哪**）
   - `dimensionId = "other"`
3. 系统确保 `other` 时间账户存在（若无则懒创建，`kind = systemHidden`，不在主页展示但在搜索/账户 Tab 可见）
4. UI 反馈：Sheet 自动收起 → 回到上一页 → Toast"'陪孩子' 的 8 个瞬间移到了'其他'"

**恢复路径：** 用户后续若添加回该时间账户信息（UC-0.2）→ 系统检查 `other` 下是否有 `originDimensionId == {被恢复的 id}` 的 Moment → 询问用户"要把 8 个原本属于'陪孩子'的瞬间收回去吗？"[是][保留在其他]

**数据变更（UC-1.3 场景）：**
- `UserProfile.children = []`
- `Dimension[id=kids].status = hidden`
- `Moment[dimensionId=kids, status=normal].*` → `dimensionId = other` + `originDimensionId = kids`

### 保存流程

1. 用户点「存入时间银行」 → 按钮变 loading 态（旋转光点）
2. 系统按顺序执行：
   - a. 创建 `Moment` 实体（UUID、各字段、createdAt=now）
   - b. 创建沙盒文件夹 `Documents/TimeBank/moments/{moment.id}/`
   - c. 遍历每个媒体项，顺序：
      - 从 PhotosPicker item 读 data
      - 写入 `{moment.id}/01.heic`（或 `.mov`）
      - 生成缩略图 `01.thumb.jpg`（200×200）
      - 创建 `MediaItem` 实体并关联到 Moment
   - d. `modelContext.save()`
3. 成功 →
   - 按钮状态消失
   - 触发**签名动效**（光点从表单中心飞到右上角，见设计规范 §6）
   - 模态自动关闭（从上向下滑出）
   - 返回到进入前的页面（主页 or 时间账户详情）
   - 顶部显示 Toast `存下了。`（1.5s 后自动消失）
   - 对应时间账户卡片的"已存入 Xh"**数字 roll-up 动画**（X → X + 刚才的时长）

### 异常

| 异常 | 反馈 |
|------|------|
| 磁盘空间不足（媒体大）| 保存失败 → 回滚所有已写文件 + Moment 插入 → Alert"手机空间不够了。清理一些再试？"[打开系统设置][取消] |
| 媒体读取/写入失败 | **V1 强原子性**：立即中止，回滚所有已写文件 → Alert"第 3 张图没读成功，整条瞬间没存下来。要不要重试？"[重试][取消] → 不允许跳过失败 |
| 用户中途杀进程 | 下次启动时 `Documents/TimeBank/moments/` 里有无对应 DB 记录的 **孤儿文件夹** → 自动清理；已插入但无 mediaItems 的 Moment 同理清理 |
| 用户返回（`×` 或手势下滑）| 若表单有内容 → 弹 Alert "先不存吗？已写的内容会丢。[取消][不存]" |
| 保存超过 10s | 按钮状态持续 → 10s 后 Toast"保存比想象中慢了一点，再等等" |

## UC-3.8【D】删除单个时刻

**入口：** 时刻详情页底部「🗑 删除」 或 时间线左滑「删除」

**步骤：**
1. 点「删除」 → 系统 **Alert**：
   ```
   确定删除这个时刻？

   存入的内容（包括 5 个媒体）会从这里消失。
   你的手机相册里不受影响。

   [取消]  [删除]
   ```
2. 点「删除」 → 执行"延迟删除 + Undo"协议（定稿 Q6）：
   - **立即**：`Moment.status = pendingDelete`（SwiftData 标记，不真正物理删除）
   - 详情页自动返回时间线 → 被删条目**淡出 + 上移动画**（300ms）
   - 时间账户卡片"已存入 Xh" **roll-down 动画**（排除 pendingDelete 的聚合结果）
   - 底部弹 Toast：`删掉了。  [撤销]`（5s 持续显示）
3. **若用户在 5s 内点「撤销」**：
   - `Moment.status = normal` 恢复
   - 时间线上该条目**重新插入 + 淡入动画**（300ms）
   - 时间账户卡片数字 roll-up 回原值
   - Toast 消失
4. **若用户未点撤销（5s 后）**：
   - 触发真正删除：`FileManager.removeItem(at: momentDir)` + `context.delete(moment)` + save
   - 物理文件夹 `moments/{moment.id}/` 彻底消失

**数据变更：**
- 5s 内：仅状态标记
- 5s 后：物理文件夹 + Moment + MediaItem（cascade）全部删除

**边界：**
- App 进入后台期间 pendingDelete 也算时间（即用户后台超过 5s 再回前台，Undo Toast 已消失 + 真正删除已完成）
- 用户杀进程：启动时对所有 `pendingDelete` 状态的 Moment 立刻执行真正删除（不可恢复）
- 连续删 N 条：Undo Toast 显示"删掉了 N 个 [全部撤销]"，点一次全部恢复

# 附录 C：关键边界与异常汇总

| 场景 | 处理 |
|------|------|
| 新增时刻时磁盘满 | 回滚 + Alert 引导打开系统设置 |
| 删除时刻中途杀进程 | 下次启动扫描孤儿文件夹并清理 |
| 移动/重命名自定义时间账户 | 所有关联 Moment 保留 id 绑定，不会丢 |
| iCloud 同步冲突 (V1.5) | 以本地为准，云端版本保留为 `v2` 后缀 |
| 用户从 iOS 系统设置直接删除 App | 所有沙盒数据丢失（符合预期），用户应导出备份 |
| 视频太大（> 500MB） | 保存时警告"这个视频 612MB，占手机空间较大。要继续存入吗？" |
| 9 宫格已满再点「+」 | 「+」按钮置灰 + Tooltip"每个时刻最多 9 个媒体" |
| 笔记文字超过 10,000 字 | 允许（不限长），但输入过程中 > 3,000 字时提示"文字挺长的，会保存得很好"|
| 用户快速连续点"保存" | 按钮 loading 态期间不响应二次点击，避免重复写入 |
| 网络断开 | V1 完全离线，无影响。V1.5 CloudKit 会自动排队 retry |
