### ④ 媒体延迟 / 视频无封面 / 存入卡死（**最高优先级 · 复杂**）

**用户原话**：
> 选择存入视频的时候，前 3-4 秒照片和视频的框哪里显示没有东西，过了几秒才加载出来，加载出来后，视频也看不到封面。点击存入时间银行后直接卡死（已发你截图）

**截图**：`3a79d2d141d746f1237d40b90a8dd9ea.jpg`、`c8bd8633efaea2bf83f273e8555e3e05.jpg` —— 视频缩略格只有一个灰色播放按钮，没有视频帧封面

**根因（Codex 必须先验证）**：
1. **视频无封面**：`MomentEditorView.mediaPreview` 的判断分支里，video 在没有 `relativePath`/`thumbnailPath`（新选入未存盘时）也没有 `data` 解码路径，落到最末分支只显示一个 `Image(systemName: "video.fill")` 占位
2. **存入卡死的根因（高度怀疑死锁）**：
   - `MomentStore.save` 是 `@MainActor` 方法
   - 内部 `try fileStore.writeMedia(...)` 是同步调用
   - `FileStore.writeMedia` 内部对每个视频跑 `generateVideoThumbnail` 和 `videoDurationSeconds`
   - 这俩函数都是 `DispatchSemaphore.wait()` 阻塞当前线程等 `Task.detached { try await asset.load(.duration) }` 的结果
   - **当 wait 发生在 main thread 上**，而 AVFoundation 内部需要回 main thread 时 → 死锁；即使不死锁，主线程也被堵住几秒导致 UI"卡死"
3. **首次延迟 3-4s**：是 PhotosPicker 的正常加载，**不是 bug**，可以忽略；但加载完后视频要立刻看到封面（这是修复重点）

**修复方案**：

**A. 让 I/O 真异步**：
- `FileStore` 标记 `@unchecked Sendable`（字段全 `let`，failureInjector 生产环境为 nil）
- 新增 async 包装：
  ```swift
  func writeMediaOffMain(_ media: [PendingMedia], to momentID: UUID) async throws -> [WrittenMedia] {
      try await Task.detached(priority: .userInitiated) { [self, media, momentID] in
          try self.writeMedia(media, to: momentID)
      }.value
  }
  // 同理 writeAdditionalMediaOffMain
  ```
- `MomentStore.save` / `update` 改 `try await fileStore.writeMediaOffMain(...)` / `writeAdditionalMediaOffMain(...)`
- **保留同步版** `writeMedia` 不删，测试还在用

**B. editor 立即看到视频封面**：
- 给 `MomentEditorMediaItem` 加 `var previewThumbnailData: Data?`
- 给 `FileStore` 加 `func makeInMemoryThumbnailData(from data: Data, kind: MediaKind, fileExtension: String, maxPixelSize: Int = 400) async -> Data?` —— 内部把 data 写到 `FileManager.default.temporaryDirectory/UUID/source.ext`，调 `generateThumbnail` 生成 `thumb.jpg`，读出 Data 返回，最后 `defer { try? FileManager.default.removeItem(...) }` 清理
- `MomentEditorView.loadPickedItems` 在每个 item append 后异步调 `generatePreviewThumbnail(for:)`，结果赋给 `draft.mediaItems[index].previewThumbnailData`
- `mediaPreview` 优先级：`thumbnailPath/relativePath` > `previewThumbnailData` > `data`（仅图片）> 占位 `ProgressView`（不再是 `Image(systemName: "video.fill")`）

**涉及文件**：
- `TimeBank/Utility/FileStore.swift`（@unchecked Sendable + offMain 包装 + makeInMemoryThumbnailData）
- `TimeBank/Shared/MomentStore.swift`（save/update 改用 await offMain）
- `TimeBank/Features/Moment/MomentEditor/MomentEditorDraft.swift`（加 previewThumbnailData 字段）
- `TimeBank/Features/Moment/MomentEditor/MomentEditorView.swift`（mediaPreview 分支 + loadPickedItems 后异步生成 thumbnail）

**Codex 重点验证**：
1. `@unchecked Sendable` 安全性：`FileStore` 字段 `baseURL`/`fileManager`/`failureInjector` 全是 `let`，无可变状态，生产 `failureInjector=nil`，仅测试注入 → **判定安全**
2. 真机测试视频选入：
   - 是否 1-2 秒内出现封面
   - 点"存入时间银行"后主线程是否还会被堵死
3. Sendable warning 是否被触发；如有，可加 `@preconcurrency` 或本地 `@unchecked` 处理

**风险**：
- 临时缩略图占内存，9 张视频 × 400px 缩略图 ≈ 数 MB，可接受
- `Task.detached` 内捕获 `self`（FileStore） strong ref，task 跑完即释放，无循环引用

### ⑦ 首页 Kh 看不懂

**用户原话**：
> 首页的时间 kh 有点奇怪，都用小时呈现就好，kh 改成中文的小时

**截图**：`5f13ce57b7094c359eb6dd930a51b1df.jpg` —— 陪孩子详情页大数字 "19.6Kh"

**根因**：
- `Formatter.hoursCompact` 当 h ≥ 10000 时返回 "X.XKh"
- 用户对 Kh（千小时）单位不熟悉

**修复方案**：
- 改 `hoursCompact`：
  - `< 10000` → `"N 小时"`（带 grouping，例 "552 小时"、"1,240 小时"）
  - `10000 ~ 1000000` → `"X.X 万小时"`（例 "1.8 万小时"）
  - `≥ 1000000` → `"NNN 万小时"`（例 "100 万小时"）
- 同步改 `hoursWithMinutes`：`"30m"` → `"30 分钟"`、`"2h"` → `"2 小时"`、`"2h 30m"` → `"2 小时 30 分钟"`
- `storedDuration` 复用 hoursWithMinutes，empty case `"0h"` → `"0 小时"`
- `lifespanSubtitle`：原本 `"45 年 · 473 Kh"`，改成 `"45 年 · X.X 万小时"`，内部转 `hoursK*1000` 喂给 `hoursCompact`

**测试同步**：
- `TimeBank/TimeBankTests/DataLayerTests.swift` `testFormatterMatrixCoversAllInterfaces`
  - 7 处断言要全更新（`552h`→`552 小时`、`18.2Kh`→`1.8 万小时`、`30m`→`30 分钟`、`473 Kh`→`47.3 万小时` 等）

**涉及文件**：
- `TimeBank/Utility/Formatter.swift`
- `TimeBank/TimeBankTests/DataLayerTests.swift`

**风险**：
- ⚠️ **PRD §21 Formatter Matrix 是否定死了 "Kh" 单位**？Codex 必须先读 `PRD-时间银行-V1.md` §21 / §22.3.1 再决定是否动 Formatter

### ⑧ 自由维度首屏 0h Bug

**用户原话**：
> 登录后最下方的自由显示是 0h，这个应该是不对的

**无截图**（但用户描述明确）

**根因（Codex 必须先 reproduce）**：
- `FreeDimensionParams.awakeHoursPerDay` 默认 14
- `DimensionCompute.consumeHours` for free = `max(0, totalAwakeHours - sum(other 5 dimensions consumeHours))`
- 常见用户配置（partner 4h/day + create 40h/wk + sport 5h/wk + kids 30h/wk）这些"非清醒时间也覆盖的活动"加起来日均 ≈14.7h，超过 awake 14h → free 残差 ≤0 → max 截断为 0

**修复方案**：
- **临时缓解**：`FreeDimensionParams` 默认 `awakeHoursPerDay: 14` → `16`
- 同步修 `DimensionParameterEditorView` 中硬编码的 `freeAwakeHoursPerDay = 16.0` 和 `restoreDefaults` 里的 `freeAwakeHoursPerDay = 16` —— 改成 `FreeDimensionParams().awakeHoursPerDay` 单一来源（避免 model 默认和 UI 默认不同步）

**已知遗留问题（不在本次范围）**：
- 公式本质上把 partner（共处时间，可能含睡眠陪伴）/ create / sport 都当成"占用清醒时间"减掉，逻辑有重叠
- 老用户 DB 已经存了 14h 的话，这次 seed 默认改 16 不会自动 migrate
- 这俩 Codex 不要管，写个 TODO 注释提醒下次迭代

**涉及文件**：
- `TimeBank/Models/DimensionParams.swift`（默认值）
- `TimeBank/Features/DimensionDetail/DimensionParameterEditorView.swift`（移除 hardcode，统一从默认读）

**风险**：
- 不影响测试（默认值变化但单测没断言这个具体值）
- 老用户行为不变（DB 已存 14）
